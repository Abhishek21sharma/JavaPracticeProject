Contract Testing is the "Consultant’s Solution" to this problem.
It allows you to test the "handshake" between the Frontend (Consumer)
and the Backend (Provider) without needing both systems to be running at the same time.

The Provider (Backend) promises to deliver data in a specific format (e.g., a JSON with id as a number).
The Consumer (Frontend/UI) promises to send requests in a specific format.
Contract testing verifies that both parties still agree on this contract.
If the Backend team tries to change id to a string, the contract test fails before the code even reaches a shared environment


Some of NFT concepts:
Load Testing: How does the system behave under "normal" expected user load?
Stress Testing: At what point does the system break (the "breaking point")?
Endurance (Soak) Testing: Does the system leak memory if it runs for 10 hours straight?
Scalability: If we double the users, does the response time stay the same?

Response Times: This is the amount of time it takes for a system to respond to a specific request.
Throughput: Throughput measures how much work the system can handle in a given time unit.Usually measured in Requests Per Second (RPS) or Transactions Per Minute (TPM).
Error Rates: This is the percentage of requests that resulted in an error (e.g., HTTP 500, 503, 504) compared to the total requests.

Data Management issues ? How to manage data in shared ENV ?
Managing data in shared environments is a classic challenge for an SDET. When multiple automation suites, manual testers, and developers hit the same environment,
you run into the "Polluted Environment" problem: Test A deletes a user that Test B was about to use, causing a "flaky" failure
that isn't actually a bug.
Manual issues:
Data Integrity & Environment Stability
Dependency & Integration Blocks : Upstream/Downstream Lack, State Conflict
Execution & Feedback Loop Issues


Automation Solution to solve this:
The "Unique Seed" Pattern: Instead of using a hardcoded user like test_admin, you generate a unique entity for every single test execution.
Implementation: Append a timestamp or UUID to names and emails (e.g., user_1714468200@test.com).
Benefit: Tests can run in parallel without ever seeing each other’s data.
Trade-off: You must ensure your system can handle a high volume of created records.

(Setup/Teardown) method:
Before: Call an API to create the exact state needed.
Test: Run the Playwright/Automation logic.
After: Call an API to delete or archive the data.

Data Pooling (The "Locker" System)
If your application makes data creation very slow (e.g., creating a Bank Account takes 30 seconds), you use a pool of pre-created accounts.
Logic: A test "checks out" a record from a database or JSON file, marks it as In_Use, and "checks it back in" when finished.

In shared environments, teardowns often fail if the test crashes. This leads to "Data Bloat."
The Janitor: Create a standalone script (or a GitHub Action/Jenkins Job) that runs on a schedule (e.g., every night at midnight).
The Logic: DELETE FROM users WHERE email LIKE '%@test-automation.com' AND created_at < NOW() - INTERVAL '24 hours'.

challenges in parallel execution ??
Resource Contention & Data Collisions
--Shared User Accounts:If Test A and Test B both log in with admin@test.com, Test B might change the password or update a profile setting that causes Test A to fail.
--Unique Constraints:If two tests try to create a record with the same "Employee ID" or "Email" simultaneously, one will hit a database conflict.
--Environment Saturation: Running 50 browsers against a single staging server can overwhelm the CPU or memory of the application under test, leading to 504 Gateway Timeouts that wouldn't happen in sequential runs.

Non-Deterministic Infrastructure (Flakiness)
--Parallelism puts massive stress on your infrastructure, whether it's local, a Grid, or a CI/CD runner.
--CI Runner Limits: GitHub Actions or Jenkins nodes have limited CPU/RAM. If you "over-shard" (e.g., 20 workers on a 2-core machine), the browsers will compete for resources, leading to extremely high TimeoutError rates.

The "Shift-Left" Approach
As an experienced tester, your greatest value is often in Shift-Left testing.
This means being involved in the design phase to catch logic flaws
 before a single line of code is written. Fixing a requirement bug costs significantly less
 than fixing a production bug.

Definition of ready:
it act as a contract between technical team and product owner.
It specifies the criteria a User Story must meet before it can be pulled into a Sprint for execution.
DoR checklist:
While every team defines its own DoR, a typical checklist for a tester approaching a new requirement includes:
Clear Title & Description: The "As a [user], I want [action], so that [value]" format is well-defined.
Acceptance Criteria (AC): Specific conditions that must be met for the story to be marked as "Done."
Visuals/Wireframes: If there is a UI component, the design or mockups are attached.
Dependencies Identified: Any third-party APIs or internal services required are available or accounted for.
Technical Feasibility: The team has discussed the story and believes it can be implemented.
Performance/Security Requirements: Non-functional requirements are explicitly stated if applicable.
